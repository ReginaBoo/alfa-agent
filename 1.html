📊 Дашборд
⚡ Raw временно в PostgreSQL, в перспективе — выгрузка в S3/MinIO
1 Identity
▼
🔐 Роль: Входная дверь. Управляет аутентификацией, сессиями и OAuth-токенами для доступа к Jira и GitHub.
  • users
    Пользователи системы — email, имя, аватар, дата регистрации
  • sessions
    Активные сессии — токен, срок действия, тип клиента (web/desktop)
  • integration_tokens
    OAuth-токены для Jira/GitHub — access_token, refresh_token, привязка к инстансу (cloud_id)
  • user_settings
    Настройки пользователя — режим "не беспокоить", автозапуск, тема
🔑 Основа всей системы
2 Core
▼
🏗️ Роль: Организационная структура. Хранит проекты и права доступа к ним.
  • projects
    Проекты внутри платформы — название, описание, владелец
  • user_projects
    Связь пользователей с проектами — роль (owner/manager/viewer)
📁 Центральная сущность
3 Integrations
▼
🔌 Роль: Переводчик. Связывает наши проекты с внешними системами (Jira, GitHub).
  • project_integrations
    Привязка проекта к Jira-проекту, GitHub-репозиторию или Confluence-пространству — external_id, type, token_id
🌉 Мост к внешним API
4 Raw
▼
📦 Роль: Страховка и отладка. Хранит сырые JSON из API Jira/GitHub.

⚠️ Временное решение: На старте храним в PostgreSQL. При росте объёмов — выгружаем в объектное хранилище (S3/MinIO).
  • raw_events
    Сырые события — source (jira/github), event_type, payload (полный JSON), timestamp
💾 Immutable storage
5 Normalized
▼
🗂️ Роль: Кеш внешних данных. Структурированное хранение задач и коммитов для быстрых запросов.
  • jira_issues
    Задачи из Jira — issue_key, статус, исполнитель, story_points, оценки времени
  • git_commits
    Коммиты из GitHub — sha, автор, сообщение, jira_issue_key (из парсинга), изменения
  • git_pull_requests
    Pull Requests — состояние, ревью, время до мержа (для метрик)
⚡ Быстрый доступ к данным
6 Analytics
▼
📈 Роль: Витрина для дашборда. Хранит предрассчитанные метрики.
  • project_metrics
    Метрики проекта — коммиты, задачи, активные пользователи, velocity, SLA
  • user_metrics
    Метрики сотрудника — нагрузка (load_score), перегруз (is_overloaded), завершённые задачи
  • project_health
    Общее здоровье проекта — health_score (0-100), статус (OK/Risk/Critical)
🎯 Готово для дашборда
7 TimescaleDB
▼
📉 Роль: Хранилище для графиков и трендов. Оптимизировано под временные ряды (расширение PostgreSQL).
  • metrics_timeseries
    Исторические значения метрик во времени — workload_index, activity_score, code_churn, sla_score
⏱️ Автоматическое партиционирование + сжатие
8 AI & Notifications
▼
🤖 Роль: Интерфейс взаимодействия с пользователем. Чаты, выводы AI, уведомления.
  • chat_sessions
    Сессии диалога с AI — привязка к пользователю и проекту
  • chat_messages
    Сообщения в чате — роль (user/assistant), текст, метаданные
  • ai_insights
    Выводы AI по проекту — тип (качество/сроки/нагрузка), текст, важность
  • notifications
    Уведомления пользователей — тип, приоритет, прочитано/скрыто
🔔 Оповещения + AI-аналитика
💡 Как внедряем: Схемы вводятся в разработку поэтапно — сначала Identity + Core, затем Integrations, Raw, Normalized, Analytics, TimescaleDB, AI.
Каждый этап проверяется на практике, БД дорабатывается по мере необходимости.
📌 Raw — временное решение. При росте объёмов выгружаем в объектное хранилище (S3/MinIO).
